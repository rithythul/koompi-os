App Permissions

Privacy

Per-app capability grants and sandboxing tiers for conventional GUI apps — Firefox, Telegram, GIMP, the software you already know. This is the legacy-app counterpart to §11 Tool Approval: §11 governs the AI agent earning tool calls; §42 governs ordinary apps earning OS capabilities. One mechanism, two consumers. The default is deny. Every grant is visible, scoped, time-stamped, and revocable from a single surface — closing the gap the §11 audit flagged: standing app grants that nothing could review or revoke. Camera, mic, location, files, network, removable media: your data is never assumed-available.

Sandboxing Tier
How much the OS can mechanically guarantee — mapped to the real Wayland/portal stack
SANDBOXED
Flatpak + bubblewrap. No host FS, no host sockets. Every capability flows through xdg-desktop-portal. The OS can prove what it cannot reach. Telegram, GIMP.
PORTAL
Native + PipeWire/portal. Runs on the host but camera, mic and screen-capture are brokered through PipeWire — device access is per-grant, not ambient. Firefox.
UNCONFINED
Vendor .deb / AppImage. Full ambient authority — we cannot enforce, only observe and warn. KOOMPI labels honestly rather than pretending. Zoom.
The badge never lies about confidence. We don't paint an unconfined app green just because the user granted it — the tier reflects the mechanism, the matrix reflects the grant.

Deny is the floor KOOMPI

macOS and Windows ship apps with broad default entitlements and ask you to claw them back. KOOMPI inverts it: every app — sandboxed or not — starts at zero and earns each capability with one explicit, scoped grant. Even an unconfined .AppImage begins with nothing granted.

§42 apps vs §11 agent

One consent engine, two consumers. §11 Tool Approval governs the AI agent asking to call a tool; §42 governs ordinary GUI apps asking for OS capabilities. Sharing the engine is why the §11-audit gap — no surface to review or revoke standing app grants — is closed here, not patched separately.

Honest tier badges

The sandbox badge reflects the mechanism (bubblewrap, PipeWire/portal, or nothing), never the grant. An unconfined app is never painted green just because you trusted it. We'd rather tell you we can only observe than pretend we can enforce.

Grants don't outlive the code Lifetime

When an app's binary hash changes on update, its standing grants pause for re-consent — the permission was given to that code, not a name. Unused grants decay to a review queue after 90 days. Across decades, the permission set stays a true picture of what you actually rely on.

Khmer-native consent

System prompts speak grammatically formal Khmer with correct register — សុំ (a request), not an imperative — because asking permission in a borrowed register is its own kind of disrespect. Consent is the most important sentence the OS says; it says it properly.

Revoke means revoke — live

Revoking a standing grant doesn't just flip a database row; it signals the running app through the portal so an in-progress mic stream actually closes. A revocation the app can ignore until next launch isn't a revocation. USB and removable media are treated as egress vectors and logged to the §28 Egress Ledger.
