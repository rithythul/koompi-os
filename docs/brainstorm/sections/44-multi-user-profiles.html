Profiles & Sharing

Isolation

A KOOMPI machine is usually one person's — but plenty are shared: a family laptop, a classroom device, a hand-me-down. The personal machine is the primary deployment; shared use is a first-class case the design handles from day one (prd §2). The OS that thinks about you must never think about you to someone else. Every profile is a cryptographic island: its Memory Palace, its cognitive inferences, its file index — sealed under a key only that person's credential can derive. Isolation here is not a feature. It is a privacy primitive.

8 · Coercion case — no deniability theater
Threat: someone forces the device open
An earlier draft offered a hidden profile + decoy passphrase — "a second login that opens a plausible empty life." It is cut, on the same grounds as §37 and §41 Backup.
✗ A hidden account and a decoy volume are forensically visible to competent examination
✗ Fake deniability is worse than none — it fails exactly when it matters
✗ A discovered decoy can turn a search into an obstruction charge
✓ Instead: per-user FDE at rest (the islands above) protects a powered-off, confiscated device
✓ Lockout throttling on unlock attempts; nothing readable while sealed
No software protects you once an adversary knows it exists. KOOMPI does not ship protections that only work if the adversary is careless.
Isolation is cryptographic, not just permissions Primitive

Filesystem permissions stop a curious sibling; they don't stop root or a stolen disk. Each profile's Memory Palace and contextd index are sealed under a key derived from that user's credential via Argon2id bound to the TPM. At rest, every other profile — including the OS itself — sees only ciphertext. This single decision makes guest-wipe, the memory boundary, and the hidden profile one coherent guarantee instead of three drawings.

The OS thinks about you, never to someone else

macOS Fast User Switching and GNOME both share a system-wide assistant and search index — your Siri suggestions and tracker results can bleed across logins. KOOMPI runs koompi-assistantd and koompi-contextd per-UID. There is no global memory to leak from. A second user's AI greets a cold context that has never heard of you.

Guardrails are rules, not wiretaps

A child profile enforces screen-time and content posture in-kernel (eBPF), but the parent dashboard exposes aggregate time and categories — never message contents. Dara's chats with the local Khmer tutor stay private to Dara. We refuse the surveillance-parenting default: a Lifetime OS that raises a child must earn that child's trust too.

Shared devices are first-class, not the default Supported case

Single-user is the primary deployment, but a shared machine — a family laptop, a school device — is handled from day one. Base models are shared by copy-on-write dedup (counted once in storage), so multiple profiles don't mean multiple full downloads. Yet each person's memory stays an island. A device shared over years ages gracefully — a child profile graduates to standard at 13 without losing a byte of their own history.
