The Gardener

Futures System

A hundred-year OS cannot wait for a vendor to notice it's sick. The local agent reads its own logs, diagnoses its own daemons, and proposes its own patches — every self-change staged on a BTRFS snapshot, shown to you as a diff, and inert until you approve. This is dangerous if done wrong, and the docs say so in plain words. It is also the honest endpoint of an OS that thinks: a system that tends itself like a garden.

Self-repair behind a snapshot, always KOOMPI

No self-change touches the live system. Diagnose → patch → sandbox replay → BTRFS snapshot → human-readable diff → your approval → staged activation with auto-rollback. Six gates between an idea and your root filesystem.

Fleet learning without surveillance

Upstreamed fixes carry code and crash signatures, never user data — counted by ZK aggregation so the project learns "this bug is everywhere" without learning where. Every machine's garden feeds the seed bank.

Dangerous, and documented as such

An OS that patches itself can wound itself; the docs say this on page one rather than in a footnote. The defense is not confidence but architecture: snapshots, sandboxed proof, a declinable diff, and a garden log that remembers every failure.
