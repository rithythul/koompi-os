Memory Etiquette

Futures RULE · v1

A conversation belongs to more than one person, so the v1 default is conservative: other people's content is excluded from the index by default. When koompi-assistantd captures a session, spans it attributes to a third party are not embedded and not written as derived facts unless the owner explicitly opts them in. The exclusion happens at write time inside the assistant daemon — not as an after-the-fact filter — so nothing about a guest ever enters the searchable store to begin with.

Write path · koompi-assistantd · third-party content is dropped before indexing default deny
assistantd capture · session 19:32
├─ attributor tag spans by speaker (owner vs. other)
├─ owner spans embedded + indexed as normal
├─ other spans NOT embedded · NO derived facts · default
└─ opt-in owner may include a party explicitly · logged

The rule is a default, not a promise about anyone else's machine: content the owner did not author is simply never handed to the embedder. Because the exclusion is enforced at write time, there is nothing to redact later and nothing to leak.

Excluded at write time, not filtered after KOOMPI

Every memory platform treats whatever the user hears as the user's data. KOOMPI's v1 default is the opposite: other people's content never enters the index, enforced by koompi-assistantd when the record is written.

A conservative default, honestly scoped

The rule governs this machine's own index — the one thing the OS can actually enforce. Including a third party is an explicit, logged opt-in, never the silent default.
