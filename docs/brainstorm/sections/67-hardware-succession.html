Ship of Theseus — Hardware Succession

Futures 1.x

Your identity lives in the Memory Palace and your keys, not the chassis. So moving to a new machine is a Lifeboat restore, not a bespoke migration tool — and at 1.x that restore path is guarded by a CI test that restores a sealed export onto fresh hardware and asserts the recovered state matches byte-for-byte. If restore-onto-a-new-machine ever breaks, the build goes red.

CI · restore onto a new machine
ci/lifeboat-restore-fresh-hw · runs on every koompi-restore change
koompi-restore import --into /dev/fresh-target
├─ target clean VM image · no prior KOOMPI state
├─ source sealed export (memory palace · keys · postures)
├─ keys re-wrapped to the target TPM
├─ vectors embed_model_id preserved per record
└─ assert recovered state == source · byte-exact

Every vector carries its embed_model_id, so a record recalled on the new machine is read with the model that made it — recall stays era-correct across an OS reinstall or a hardware swap. The self is the data plus the keys; the CI test proves the data plus the keys actually move.

Restore is a tested path, not a promise KOOMPI

"Your data is portable" is only true if it's exercised. The Lifeboat restore path is verified by a CI job that restores a sealed export onto fresh hardware and diffs the result — so buying a new laptop is a supported, tested operation, not a hopeful one.

Identity survives the box

Keys re-wrap to the new machine's TPM and vectors keep their embed_model_id, so recall works the same after the move. The chip changed; the self did not.
