The Continuity Charter

Futures System

You're being asked to trust this OS for fifty years. Companies rarely live thirty. So the hardest promise is written down, in public, with machinery behind it: if KOOMPI the company dies, KOOMPI the OS does not. Everything is reproducible from public repos, the signing keys are escrowed behind a dead-man's switch, and the protocol is specified so thoroughly that strangers in 2060 could re-implement it from the documents alone.

"What if KOOMPI dies?" — the written answer, as a timeline
Day 0 — KOOMPI ceases operations
Every machine keeps working in full. Updates pause; nothing else changes. The escrow heartbeat stops.
Day 30 — council confirms silence
Keyholders independently verify the company is gone, not merely offline. Public notice posted; 60-day clock continues.
Day 90 — keys released, 3-of-5
The dead-man's switch completes. The council can now sign updates against the public repos. Machines accept them because the keys are the same keys.
Day 90+ — community succession
Maintainership passes to the community under the charter's governance. To a user in Battambang or Berlin, nothing visible happened — which is the entire point.
The promise no Big Tech OS can make KOOMPI

Ask any platform "what happens to my OS if you die?" and you get silence — their business depends on you not asking. KOOMPI publishes the answer, with status checks you can verify today. The charter is a trust weapon precisely because it's falsifiable.

The spec outlives the code

The memory schema, egress format, and capability bus are documents first, implementations second. A team in 2060 with no KOOMPI source could rebuild a compatible OS from the spec — the same way TCP/IP outlived every original stack.

Continuity is enforced, not pledged

Goodwill doesn't survive bankruptcy courts; key shards and public repos do. The dead-man's switch turns "we promise" into a mechanism that executes even if nobody at KOOMPI is left to keep the promise.
