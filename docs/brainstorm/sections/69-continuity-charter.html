The Continuity Charter

Futures v1

The Continuity Charter is a real document, committed to docs/, that answers one question in writing: if KOOMPI the company dies, what happens to your machine? Its answer is "nothing you can see" — because every package builds from public repos, the memory and egress schemas are specified independently of the code, and your machine never phones home for permission to run. It's not a pledge in a blog post; it's a versioned file with guarantees you can check against the repo today.

in repo
§1
Reproducible from public source. Every package builds from public repos. No private build step is load-bearing.
§2
Protocol over implementation. The memory schema and egress format are specified independently of the code, so anyone may re-implement them. The spec is the contract.
§3
Your machine never depends on us being alive. No phone-home, no license check, no kill switch. A machine that never hears from KOOMPI again keeps working (ADR-0001: degrade local, never silent cloud).
Each clause maps to something checkable in the repo — public package sources, a versioned schema file, and the ADR-0001 no-phone-home invariant. The charter is a trust weapon precisely because it's falsifiable against the code that ships.
The promise, written down and committed KOOMPI

Ask most platforms "what happens to my OS if you die?" and you get silence. KOOMPI's answer is a file in the repo: build from public source, spec over implementation, no phone-home. You can read it and check it today.

The spec outlives the code

The memory schema and egress format are documents first, implementations second — so a compatible OS can be rebuilt from the spec even with no original KOOMPI source, the way TCP/IP outlived every original stack.

Shelved

The full governance apparatus — a seated cross-border council, 3-of-5 key escrow, and a 90-day dead-man's switch that auto-releases signing keys — is described in the charter as intent but stays on the horizon shelf (horizon.md) until it can be seated and audited for real.
