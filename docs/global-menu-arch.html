Technical Design Document: Rust-Powered Global Menu for Quickshell

A native bridge between Linux application D-Bus menus and a Quickshell top panel, for the KOOMPI desktop session.

Version1.1
Date2026-06-12
StatusDraft
Target Stack Rust (ADR-0004) Quickshell 0.1 Hyprland D-Bus Wayland

1. Overview

Most Linux desktop applications already expose their menu bars over D-Bus using the com.canonical.AppMenu.Registrar and com.canonical.dbusmenu protocols — the same mechanism that powered Unity's global menu and KDE Plasma's "Application Menu" widget. A GTK or Qt application registers its top-level window with the registrar, exporting an object path that holds the full, live menu tree.

The gap is the compositor. Under Wayland, compositors do not propagate this menu registration to the shell. There is no standard surface protocol that hands the active window's menu to a panel. Each shell has to discover and assemble it independently.

This design closes that gap with a small native Rust daemon (ADR-0004), koompi-globalmenu, which:

  1. Watches the active window through the compositor's IPC socket (Hyprland's .socket2.sock event stream).
  2. Looks up that window's D-Bus menu registration via GetMenuForWindow.
  3. Fetches the complete menu tree with GetLayout on the application's dbusmenu object.
  4. Serializes the tree to a compact JSON line and streams it to Quickshell over stdout.

On the shell side, Quickshell launches the daemon with Io.Process, parses each JSON line, and feeds the result into a DBusMenu-style rendering chain that draws the menus in the top panel bar — producing a macOS-style global menu that always reflects the focused application.

3. Rust Daemon Specification

3.1 Functional Requirements

  1. Connect to the Hyprland IPC socket identified by $HYPRLAND_INSTANCE_SIGNATURE.
  2. Subscribe to activewindow and activewindowv2 events.
  3. On each active-window event, extract the window address and PID.
  4. Query com.canonical.AppMenu.Registrar.GetMenuForWindow(windowId) → (serviceName, menuObjectPath).
  5. Introspect the DBusMenu at that path: GetLayout(parentId=0, recursionDepth=-1, propertyNames=[]) → full menu tree.
  6. Serialize the tree to JSON and emit it to stdout.
  7. On focus change, emit a new payload; on an app without a menu, emit {"menus": null, "appName": "…"}.
  8. Handle D-Bus errors gracefully and reconnect on socket failure.

3.2 D-Bus Mapping Logic

Resolving a window's menu is a two-step D-Bus lookup. First, the registrar translates the window ID into the service name and object path of whichever process owns the menu (GetMenuForWindow). Second, that service is queried directly with GetLayout to retrieve the menu tree. The structs below model the parsed result and the registration call.

rust
use std::time::Duration;

use serde::Serialize;
use zbus::{zvariant::OwnedObjectPath, Connection};

#[derive(Serialize)]
#[serde(rename_all = "camelCase")]
pub struct MenuNode {
    pub id: i32,
    pub label: String,
    pub enabled: bool,
    pub visible: bool,
    pub children: Vec<MenuNode>,
    pub separator: bool,
}

#[derive(Serialize)]
#[serde(rename_all = "camelCase")]
pub struct MenuPayload {
    pub app_name: String,
    pub window_id: u64,
    pub service: String,
    pub object_path: String,
    pub menus: Option<Vec<MenuNode>>,
}

// Method call: com.canonical.AppMenu.Registrar
//              /com/canonical/AppMenu/Registrar
//              com.canonical.AppMenu.Registrar.GetMenuForWindow
#[zbus::proxy(
    interface = "com.canonical.AppMenu.Registrar",
    default_service = "com.canonical.AppMenu.Registrar",
    default_path = "/com/canonical/AppMenu/Registrar"
)]
trait AppMenuRegistrar {
    fn get_menu_for_window(&self, window_id: u32) -> zbus::Result<(String, OwnedObjectPath)>;
}

/// Query AppMenu.Registrar for the menu service/path of the focused window.
pub async fn get_menu_registration(
    conn: &Connection,
    window_id: u64,
) -> anyhow::Result<(String, String)> {
    let registrar = AppMenuRegistrarProxy::new(conn).await?;

    // Parse (s, o) reply → service name + object path (2000 ms reply timeout)
    let (service, path) = tokio::time::timeout(
        Duration::from_millis(2000),
        registrar.get_menu_for_window(window_id as u32),
    )
    .await
    .map_err(|_| anyhow::anyhow!("no registrar reply within 2000 ms"))??;

    Ok((service, path.to_string()))
}

3.4 Hyprland IPC Event Loop (Rust)

The daemon connects to Hyprland's event socket (.socket2.sock) and reads newline-delimited eventname>>data records. On each activewindow event it resolves the window address, performs the D-Bus lookup, and writes the serialized payload to stdout.

rust
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use tokio::net::UnixStream;

pub async fn connect_hyprland_ipc() -> anyhow::Result<UnixStream> {
    let sig = std::env::var("HYPRLAND_INSTANCE_SIGNATURE")
        .map_err(|_| anyhow::anyhow!("Hyprland not running"))?;
    let xdg_runtime = std::env::var("XDG_RUNTIME_DIR")
        .unwrap_or_else(|_| "/run/user/1000".to_string());
    let socket_path = format!("{xdg_runtime}/hypr/{sig}/.socket2.sock");
    Ok(UnixStream::connect(socket_path).await?)
}

pub async fn event_loop(stream: UnixStream, conn: &Connection) -> anyhow::Result<()> {
    let mut lines = BufReader::new(stream).lines();
    let mut stdout = tokio::io::stdout();

    while let Some(line) = lines.next_line().await? {
        // Events arrive as "eventname>>data\n"
        if let Some(data) = line.strip_prefix("activewindow>>") {
            // data = "windowClass,windowTitle"
            let _window_class = data.split(',').next().unwrap_or(""); // use for appName fallback
            // Get window address for D-Bus lookup
            let addr = get_active_window_address().await?;
            let window_id = u64::from_str_radix(&addr, 16)?;
            let payload = match fetch_menu_payload(conn, window_id).await {
                Ok(p) => p,
                Err(err) => {
                    eprintln!("menu fetch failed: {err}");
                    MenuPayload {
                        app_name: "Unknown".to_string(),
                        window_id,
                        service: String::new(),
                        object_path: String::new(),
                        menus: None,
                    }
                }
            };
            let json = serde_json::to_string(&payload)?;
            stdout.write_all(json.as_bytes()).await?;
            stdout.write_all(b"\n").await?;
        }
    }
    Ok(())
}

4. Quickshell Integration Specification

4.1 Io.Process Integration

Quickshell owns the daemon's lifecycle. A single Process launches it, reads stdout line-by-line through a SplitParser, and exposes the parsed menu data as reactive QML properties. The daemon is auto-restarted on a non-zero exit.

qml
// GlobalMenuBridge.qml
import Quickshell
import Quickshell.Io
import QtQuick

QtObject {
    id: root

    // Parsed menu data from the Rust daemon
    property var currentMenus: []
    property string currentApp: ""

    // Launch the daemon and read its stdout line-by-line
    Process {
        id: menuDaemon
        command: ["/usr/lib/koompi/koompi-globalmenu"]
        running: true

        // Auto-restart if the daemon crashes
        onExited: function(code, status) {
            if (code !== 0) {
                console.warn("GlobalMenu daemon exited:", code, status)
                Qt.callLater(() => { running = true })
            }
        }

        stdout: SplitParser {
            splitMarker: "\n"
            onRead: function(data) {
                if (data.trim() === "") return
                try {
                    const payload = JSON.parse(data)
                    root.currentApp   = payload.appName  ?? ""
                    root.currentMenus = payload.menus    ?? []
                } catch (e) {
                    console.error("GlobalMenu parse error:", e, data)
                    root.currentMenus = []
                }
            }
        }
    }
}

4.2 DBusMenu Rendering in Bar

MenuBarRow binds the parsed model to a Repeater of top-level menu entries, prefixed by a bold application-name label (macOS convention). Apps without a menu render nothing in the menu zone.

qml
// MenuBarRow.qml — renders parsed menu items in the top panel
import Quickshell
import QtQuick
import QtQuick.Layouts

RowLayout {
    id: root

    required property var   menus
    required property string appName

    spacing: 0

    // App name label (macOS-style, bold)
    Text {
        text: root.appName
        font.pixelSize: 12
        font.weight: Font.Bold
        color: Appearance.colors.colOnLayer0
        Layout.rightMargin: 12
        visible: root.appName !== ""
    }

    // Dynamic menu entries
    Repeater {
        model: root.menus

        delegate: MenuBarEntry {
            required property var  modelData
            label:    modelData.label
            children: modelData.children
            enabled:  modelData.enabled
            visible:  modelData.visible && !modelData.separator
        }
    }

    // Fallback: no menu registered
    Text {
        visible: root.menus.length === 0
        text: ""     // empty — bar shows nothing for apps without menus
        color: "transparent"
    }
}

4.3 Dropdown Popup

Each top-level entry owns a PanelWindow dropdown in the Wayland overlay layer. Selecting a leaf item closes the dropdown and routes an activation back through the daemon, which issues the corresponding D-Bus Event call to the owning service.

qml
// MenuBarEntry.qml — single top-level menu item with dropdown
import Quickshell
import QtQuick
import QtQuick.Controls

Item {
    id: root

    required property string label
    required property var    children
    required property bool   enabled
    required property bool   visible

    implicitHeight: Appearance.sizes.baseBarHeight
    implicitWidth:  labelText.implicitWidth + 20

    property bool open: false

    Text {
        id: labelText
        anchors.centerIn: parent
        text: root.label
        font.pixelSize: 12
        color: root.open
               ? Appearance.colors.colOnSecondaryContainer
               : Appearance.colors.colOnLayer0
        opacity: root.enabled ? 1.0 : 0.4
    }

    // Highlight background when open
    Rectangle {
        anchors.fill: parent
        radius: 4
        color: root.open
               ? Appearance.colors.colSecondaryContainer
               : "transparent"
        z: -1
    }

    MouseArea {
        anchors.fill: parent
        onClicked: root.open = !root.open
    }

    // Dropdown popup anchored below this item
    PanelWindow {
        id: dropdown
        visible: root.open
        WlrLayerShell.layer: WlrLayerShell.Layer.Overlay
        // position relative to root item omitted for brevity

        Column {
            Repeater {
                model: root.children
                delegate: MenuDropdownItem {
                    required property var modelData
                    label:    modelData.label
                    enabled:  modelData.enabled
                    separator: modelData.separator
                    onTriggered: {
                        root.open = false
                        // Activate via D-Bus Event call
                        menuDaemon.activateItem(modelData.id)
                    }
                }
            }
        }
    }
}

5. Inter-Process Data Contract

5.1 JSON Schema (Draft-07)

The wire format is validated against the following schema; menus is the discriminator between the normal and fallback cases.

json schema
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "koompi-globalmenu-payload",
  "type": "object",
  "required": ["appName", "windowId", "menus"],
  "properties": {
    "appName":    { "type": "string",  "description": "WM_CLASS or window title for display" },
    "windowId":   { "type": "integer", "description": "Hyprland window address (hex parsed to uint64)" },
    "service":    { "type": "string",  "description": "D-Bus service name owning the menu, empty string if none" },
    "objectPath": { "type": "string",  "description": "D-Bus object path of the DBusMenu, empty string if none" },
    "menus": {
      "oneOf": [
        { "type": "null",  "description": "App has no D-Bus menu registration" },
        { "type": "array", "items": { "$ref": "#/$defs/MenuNode" } }
      ]
    }
  },
  "$defs": {
    "MenuNode": {
      "type": "object",
      "required": ["id", "label", "enabled", "visible", "separator", "children"],
      "properties": {
        "id":        { "type": "integer" },
        "label":     { "type": "string"  },
        "enabled":   { "type": "boolean" },
        "visible":   { "type": "boolean" },
        "separator": { "type": "boolean", "description": "True = render as 
, ignore label" },
        "children":  { "type": "array", "items": { "$ref": "#/$defs/MenuNode" } }
      }
    }
  }
}

5.2 Error States and Fallback Behaviors

ConditionDaemon BehaviorQuickshell Behavior
Window has no AppMenu registration Emit {menus:null} Show only appName, no menu items
D-Bus daemon unreachable Emit {menus:null, appName:"(no menu)"} Bar renders empty menu zone
Daemon crashes / exits non-zero (process exits) Io.Process.onExited restarts after 500ms
JSON parse error in QML (daemon sends malformed data) Log error, set menus=[] gracefully
Window focus changes rapidly Drops stale responses, emits only the latest Last parsed payload overwrites previous
App registers menu after spawn delay Retries GetMenuForWindow 3× with 100ms backoff Re-renders when a valid payload arrives

5.3 Security Considerations

  • The daemon runs as the user and reads only its own session D-Bus.
  • The stdout pipe is process-local; there is no network exposure.
  • JSON is parsed with try/catch in QML — malformed input cannot crash the shell.
  • Menu item activation sends a D-Bus Event call to the specific service/path only — no shell-level code execution.
  • No setuid, no capabilities required beyond normal session D-Bus access.

6. Build & Packaging (Quickshell Module)

The daemon is built as a standalone executable with cargo build --release (zbus is a pure-Rust D-Bus implementation — no dbus-1 C linkage) and installed to /usr/lib/koompi/koompi-globalmenu, the path referenced by the Quickshell Process command.
